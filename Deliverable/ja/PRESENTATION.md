### スライド 2 — 課題

**MEP 電気設計は反復的でクリック操作が多い**

- 3×4 のダウンライトグリッドを手動配置: 20 回以上のクリック、間隔が違ったら取り消しも手間
- 12 台の器具を回路に接続: パネルブラウザを開き、器具を一つずつドラッグ
- 回路の過負荷確認: スケジュールを開き、負荷を手動で合計
- 新入社員やバイリンガルチームは、電気設計の習熟よりも Revit UI の学習に時間を費やす
- コマンド名は英語表記が多く、日本語話者にはとっつきにくい

**本質的な問題**: 設計者の*意図*は一文で表現できるのに、Revit の操作は数十ステップ必要。

### スライド 3 — ソリューション

**RevitMate: 自然言語 → 安全な Revit 操作**

- Revit 内のドッキングチャットパネル — 外部ウィンドウや コピー&ペーストは不要
- 英語または日本語でコマンドを入力するだけで、Claude が計画・実行
- Claude の *ツールユース* 機能が、検証済みの Revit API 関数を呼び出す
- すべての変更操作は名前付き Revit トランザクション → Ctrl+Z で完全に取り消し可能
- 読み取り専用クエリはモデルに触れずに即座に回答

```
ユーザー:  選択した部屋にダウンライトを3×4のグリッドで配置して。
↓
Claude が呼び出す: get_room_info → create_light_fixture (×12)
↓
「✓ 12台の器具を作成しました。グリッド間隔: 1,200 mm × 900 mm」
```

### スライド 4 — ライブデモ

*(Revit に切り替えてライブデモを実施)*

**デモシーケンス**

1. 部屋を選択 → "Place a 3×4 grid of 100 W downlights"（英語コマンド）
2. 器具を選択 → 「選択中の器具をLP-1の回路3に接続して。」（日本語コマンド）
3. "Is circuit 3 on LP-1 overloaded?"（英語クエリ）
4. チャットパネル内のツール実行トレースを確認

**デモ中に強調するポイント**

- チャットボックスをクリックしても選択ピン（緑のタグ）が消えない（Win32 フォーカス奪取バグ修正済み）
- ツール呼び出しのアクションバブルがリアルタイムで表示される（紫・斜体スタイル）
- Ctrl+Z で AI の操作を1ステップずつ取り消せる

### スライド 5 — 3 層アーキテクチャ

```
┌─────────────────────────────────────────────────┐
│             UI レイヤー  (WPF)                   │
│  DockablePane · チャット · EN/JP i18n · MVVM     │
└──────────────────────┬──────────────────────────┘
                       │  ユーザーメッセージ + 選択コンテキスト
                       ▼
┌─────────────────────────────────────────────────┐
│         AI レイヤー  (RevitMate.Core)            │
│  ClaudeApiClient · ツールスキーマ · エージェントループ │
└──────────────────────┬──────────────────────────┘
                       │  tool_use ブロック (JSON)
                       ▼
┌─────────────────────────────────────────────────┐
│      エグゼキューターレイヤー  (RevitMate.Addin) │
│  ToolDispatcher · トランザクション · ExternalEvent │
└──────────────────────┬──────────────────────────┘
                       │  Revit API 呼び出し
                       ▼
                 Revit ドキュメント
```

- **Core に Revit 参照なし** — 単体テストが独立して実行可能
- **トランザクションを開けるのはエグゼキューターレイヤーのみ**
- **すべての Revit API 呼び出し**は `ExternalEvent` 経由でメインスレッドにマーシャリング

### スライド 6 — Claude ツールユースの仕組み

**`SendAsync()` 内のエージェントループ**

```
1. ユーザーがメッセージを送信
2. RevitMate → Claude: { messages, tools: [7つのスキーマ], system_prompt }
3. Claude → RevitMate: { content: [ { type: "tool_use", name: "get_room_info", ... } ] }
4. RevitMate が ExternalEvent 経由でツールを実行し、結果 JSON を取得
5. RevitMate → Claude: { tool_result: "{ area: 42.5, ... }" }
6. Claude → RevitMate: { content: [ { type: "tool_use", name: "create_light_fixture", ... } ] }
7. Claude が type: "text" を返すまで繰り返す → チャットに表示
```

**ツールスキーマの例 — `create_light_fixture`**

```json
{
  "name": "create_light_fixture",
  "description": "指定されたレベルに照明器具インスタンスを配置します...",
  "input_schema": {
    "type": "object",
    "required": ["family_name", "level_name", "grid_x_mm", "grid_y_mm"],
    "properties": {
      "family_name": { "type": "string" },
      "count":       { "type": "integer", "default": 1 }
    }
  }
}
```

- **7 つのツール**を宣言し、呼び出す順序は Claude が自律的に判断
- **選択スナップショット**をフォーカス奪取後も維持するために `get_selected_elements` の入力に注入

### スライド 7 — 成果

| 指標 | 値 |
|---|---|
| 実装ツール数 | 7 |
| ソリューション内プロジェクト数 | 4 |
| C# コード行数（概算） | 約 2,000 行 |
| 対応言語 | 英語、日本語 |
| 対応 Revit バージョン | 2026 |
| アンドゥ粒度 | ツール呼び出し単位（Revit トランザクション） |
| API キー保存方式 | Windows DPAPI による暗号化 |
| 開発期間 | 約 3 週間（1 人） |

**難しかった点**

- Win32 フォーカス奪取バグ: `SelectionChanged([])` が `WM_KILLFOCUS` 処理中に発火し、WPF イベントより先に実行される問題。`GetFocus()` P/Invoke をイベントハンドラ内で直接呼び出すことで解決。
- 非同期 Claude レスポンスを Revit API のシングルスレッドにマーシャリングするため、`ExternalEvent` と `TaskCompletionSource` を組み合わせた仕組みの構築。
